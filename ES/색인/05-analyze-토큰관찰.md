# _analyze 로 토큰 관찰

---

## _analyze 란?

---

- `POST _analyze` 로 호출할 수 있다.
- 문서를 색인하지 않고, "이 문자열을 이 analyzer 로 돌리면 어떤 토큰이 나오나" 를 즉석에서 찍어 볼 수 있다.
    - 앞으로 "왜 이 검색어가 안걸리지?" 를 디버깅 할 수 있는 무기가 된다.

## 응답필드 읽는 법

---

```json
{
  "tokens": [
    {
      "token": "맞바",
      "start_offset": 0,
      "end_offset": 2,
      "type": "word",
      "position": 0
    },
    ...
  ]
}
```

- `token` : 최종 토큰 (역색인에 들어갈 값)
- `position` : 몇 번째 토큰인지 (0, 1, 2, ...)
- `start/end_offset` : 원문에서의 글자 위치 - 하이라이트에서 사용된다.
- `type` : 토큰 종류 (\<HANGUL>, \<ALPHANUM>, word, ...)

## analyzer 실측 별 결과

---

| analyzer / tokenizer | 입력              | 결과 토큰             | 핵심                         |
|----------------------|-----------------|-------------------|----------------------------|
| standard             | [개정판] 맞바람       | 개정판, 맞바람          | 공백/부호로 자름                  |
| keyword              | [개정판] 맞바람       | [개정판] 맞바람         | 안쪼갬, type=word, offset=0~9 |
| standard             | Hello WORLD 맞바람 | hello, world, 맞바람 | lowercase 필터 동작 (대->소문자)   |
| ngram(2~3)           | 맞바람             | 맞바, 맞바람, 바람       | 조각 색인 = 부분일치 원리            |
| nori                 | 맞바람이 분다         | ❌ 에러              | 플러그인 필요                    |

## 알게 된 것 (인사이트)

---

- offset 은 "원문" 의 위치이다.
    - 그래서 offset 이 1부터면 -> 0번 글자가 제거된 것이다. 무엇이 버려졌는지 역추적이 가능하다.
    - 예) 
        - analyzer: standard / text: [개정판] 맞바람
        - => 개정판 offset 1~4 (0번 `[` 가 제거됨)
- 소문자화는 `tokenizer` 가 하는일이 아닌, `token_filter` 의 일이다.
- `ngram` : 미리 조각내어 색인한다. -> `바람` 만 검색해도 `맞바람` 문서가 걸린다.
- 검색어도 똑같이 쪼개져 역색인을 조회한다.
