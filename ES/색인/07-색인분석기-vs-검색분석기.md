# 색인 analyzer, 검색 analyzer

---

## 텍스트는 "두 번" 분석된다.

---

- 색인 시점 (analyzer) : 문서의 필드값 -> 토큰 -> 역색인 저장
- 검색 시점 (search_analyzer) : 검색어 -> 토큰 -> 저장된 토큰과 대조
- 검색 : `검색어 토큰` == `저장 토큰` 비교

```json
{
  "ngram": {
    "analyzer": "ngram_analyzer",
    "search_analyzer": "standard"
  }
}
```

- `analyzer` : 색인 시점에는 ngram 을 사용하여 조각낸다.
- `search_analyzer` : 검색 시점에는 통째로 검색한다.

## 왜 이렇게 사용하는가?

---

**"맞바람 외전"** 이라는 제목을 색인한다고 해보자.

색인 시점 (ngram_analyzer, 2~5글자) - 제목을 잘게 조각내 저장한다.

- "맞바", "맞바람", "바람", "외전", ... 등 -> 수많은 조각들이 역색인에 저장된다.

이제 사용자가 "맞바람" 을 검색한다. 검색 분석기가 뭐냐에 따라

1. 케이스 A - 검색도 `ngram` 이면 검색어도 조각난다.
    - "바람" 조각으로 인해, "바람의 검", "봄바람" 같은 무관한 문서들도 다 걸리게된다.
    - 과도한 매칭이 발생한다.
2. 케이스 B - 검색은 `standard` 만
    - "맞바람" 이 저장된 조각들 중에 있나?
        - "맞바람 외전" 은 색인때 "맞바람(3-gram)" 을 저장했으니 있다.
        - "바람의 검" 은 검색되지 않는다.


## search_analyzer 동적 변경

---

- `search_analyzer` 는 재색인 없이 변경이 가능하다.
    - 저장 역색인 데이터를 건들지 않기 때문이다.
    - `PUT _mapping` 으로 search_analyzer 를 추가하거나 변경 할 수 있다.
- `search_analyzer` 우선순위
    - 쿼리에 명시한 analyzer > search_analyzer > analyzer
