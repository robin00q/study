# 멀티필드

---

## 정의

---

```json
{
  "subject": {
    "type": "text",
    "fields": {
      "keyword": {
        "type": "keyword"
      },
      "ngram": {
        "type": "text",
        "analyzer": "ngram_analyzer"
      },
      "korean": {
        "type": "text",
        "analyzer": "nori"
      },
      "english": {
        "type": "text",
        "analyzer": "english"
      }
    }
  }
}
```

- `subject` : 루트 (text, 전문검색)
- `subject.keyword` : 정확히 일치
- `subject.ngram` : 부분일치
- `subject.korean` : 한글 검색 - 형태소 일치
- `subject.english` : 영문 검색 - 불용어 제거 + 어간추출: running → run

## 쿼리

---

### 필드 하나에 대하여 쿼리하는 방식

---

```json
// 전문검색을 원하는 경우
{
  "match": {
    "subject": "맞바람"
  }
}
```

```json
// 정확한 일치를 원하는 경우
{
  "term": {
    "subject.keyword": "맞바람"
  }
}
```

```json
// 부분일치를 원하는 경우
{
  "match": {
    "subject.ngram": "맞바람"
  }
}
```

### 필드 여러개에 대해 쿼리하는 방식

---

```json
{
  "multi_match": {
    "query": "맞바람",
    "fields": [
      "subject.korean",
      "subject.ngram"
    ]
  }
}
```

왜 사용하는가?

- `text` 는 정확일치 혹은 정렬이 불가능하다.
- `keyword` 는 전문검색이 불가능하다.
- **제목 하나로 부분검색, 정확일치, 정렬을 모두** 하려면 하나만 고르지말고 여러벌을 색인하여 사용한다.

## 비용

---

- 하위 필드마다 별도 역색인을 생성하면 디스크 부하가 증가하며, 색인시간이 증가 할 수 있다.
    - 단 원본(_source) 는 한벌이다.
    - 검색 색인이 N벌 생기는것이다.
