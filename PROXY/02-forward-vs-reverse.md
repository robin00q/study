# Forward vs Reverse 프록시

---

## Forward 프록시와 Reverse 프록시는 어떤(클라이언트/서버) 편에 서서 대리하는가?

---

- `Forward` 프록시 : `클라이언트` 편에 서서, 클라이언트가 밖으로 나가는(outbound) 요청을 대리한다.
- `Reverse` 프록시 : `서버` 편에 서서, 서버로 들어오는(inbound) 요청을 대리한다.

```text
[Forward] 클라이언트 -> [프록시] -> 인터넷의 여러 서버

[Reverse] 인터넷의 여러 클라이언트 -> [프록시] -> 백엔드 서버
```

## 프록시의 존재를 아는 쪽은?

---

- `Forward` :
    - 클라이언트는 안다. (프록시를 사용하겠다고 설정함)
    - 목적지 서버는 모른다. (그냥 프록시가 클라이언트 인 줄 앎)
- `Reverse` :
    - 클라이언트는 모른다. (그냥 서버에 접속한 줄 앎)
    - 서버(운영자) 는 안다. (자기가 앞에 세운 것)

## Forward 프록시 대표 사용 사례 3가지

---

1. 회사 아웃바운드 통제 - 직원 PC 의 외부 접속을 프록시로 모아 차단/로깅/감사
2. IP 우회/허용목록 통과 - 등록된 IP 로 나가야 하는 경우 (특정 서버 접근 제한 우회)
3. 캐싱으로 대역폭 절약 - 여러 클라이언트가 같은 외부 자원을 받을 때 프록시가 캐시
4. IP 익명화
5. 컨텐츠 필터링

## Reverse 프록시의 대표 사용 사례 3가지

---

1. 로드밸런싱 - 뒤의 백엔드 여러 대를 두고 요청을 분배
2. TLS 인증서 관리 일원화 - 프록시가 HTTPS 를 받아 복호화하고, 백엔드에는 HTTP 로 전달
3. 캐싱 / 정적 콘텐츠 제공 / 압축 - 백엔드 부하 경감
4. DDos 완충
5. WAF

## 회사 아웃바운드 프록시 vs 웹서비스 앞단 nginx

---

- 회사 아웃바운드 프록시 = Forward : 직원(클라이언트)들이 밖으로 나가는걸 통제
- 웹서비스 앞단 nginx = Reverse : 외부 사용자가 우리 서비스로 들어오는걸 서버 편에서 받으니까

## 같은 소프트웨어가 forward / reverse 도 되나?

---

된다.

- nginx / HAProxy / Envoy 등은 설정에 따라 둘 다 가능하다.

- nginx 에 proxy_pass 로 정해진 백엔드로 들어온 요청을 넘기게 설정 -> reverse
- 클라이언트가 자신을 프록시로 지정해 임의 목적지로 나가는 요청을 받게 설정 -> forward

## Transparent(투명) 프록시는 일반 forward 와 뭐가 다른가?

---

- 일반 forward : 클라이언트가 "나 이 프록시 쓸게" 라고 명시적으로 설정해야 한다. (브라우저/OS 프록시 설정, HTTP_PROXY 등)
- Transparent 프록시 :
    - 클라이언트는 프록시를 인지하지 못한다. 
    - 클라이언트는 아무 설정도 안한다. 네트워크 장비(라우터/게이트웨이) 가 트래픽을 강제로 프록시로 우회시킨다.
