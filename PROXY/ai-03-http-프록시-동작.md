# 03 · HTTP 프록시 동작 (GET 대리요청 · CONNECT 터널) — 질문

> 범위: HTTP 프록시가 평문/암호화 트래픽을 각각 어떻게 중개하나. 답은 `03-...md`(내 노트)에.

1. 평문 HTTP GET을 프록시로 보낼 때, 클라이언트가 프록시에 보내는 **요청 라인**은 직접 연결과 어떻게 다른가? (absolute-URI vs origin-form)
2. HTTPS(TLS)는 프록시가 내용을 복호화 못 하는데, 어떻게 중개가 가능한가? `CONNECT` 메서드의 역할은?
3. `CONNECT`로 터널이 열린 뒤, 프록시가 하는 일은 무엇인가? (바이트 릴레이 = 무슨 뜻)
4. 프록시가 HTTPS 내용을 들여다보려면(가로채려면) 무엇이 필요한가? (MITM, 사설 CA 인증서)
5. `Via`, `X-Forwarded-For`, `Proxy-Connection` 헤더는 각각 무엇을 나타내나?
6. HTTP 프록시로는 왜 임의의 TCP 프로토콜(예: DB 접속)을 그냥 태우기 어려운가? (`CONNECT` 허용 포트 제한과 연결지어)
7. 캐싱 프록시가 "이 응답을 캐시해도 되는지" 판단할 때 보는 HTTP 헤더는? (`Cache-Control` 등)
