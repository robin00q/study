# SOCKS 프록시

---

## SOCKS 는 어느 계층인가?

---

- `HTTP 프록시` : **L7(응용 계층)** 에서 동작. HTTP 를 이해해서 대리한다. (요청 라인 파싱, 헤더 조작, 캐싱)
- `SOCKS 프록시` : **L5(세션 계층)** 에서 동작. TCP 바로 위에서 동작. 내용을 이해하지 않고, TCP 연결 자체를 대리한다.

> HTTP 프록시 = "HTTP 요청을 대신 보내주는 대리인" / SOCKS 프록시 = "TCP 연결을 대신 열어주는 중계기"

SOCKS 는 "어디에 연결할래?" 만 듣고 TCP 통로를 뚫어준 뒤엔 바이트를 그대로 흘린다.

## SOCKS4 vs SOCKS5

---

|        | SOCKS4            | SOCKS5               |
|--------|-------------------|----------------------|
| 인증     | 없음                | 인증 지원(user/pass 등)   |
| 주소 형식  | IPv4              | IPv4 + IPv6 + 도메인 이름 |
| 도메인 해석 | 클라이언트가 미리 IP 로 변환 | 서버(프록시) 가 도메인 해석 가능  |
| UDP    | X                 | UDP 지원               |

핵심은 SOCKS5 가 인증 + IPv6/도메인 + UDP를 더한 것

## SOCKS5 는 DNS 를 어디서 하나 (remote DNS)

---

- SOCKS5 에서는 클라이언트가 IP 대신 도메인 이름을 그대로 프록시에 넘길 수 있다.
- 그러면 DNS 조회를 프록시가 수행한다.

장점

1. 프라이버시: 로컬에서 DNS를 조회하면 "내가 어디에 접속하려는지"가 로컬 네트워크/ISP에 노출됩니다(DNS 쿼리 유출). 원격 DNS면 그 조회조차 프록시 뒤에서 일어나 안 새어나갑니다.
2. 도달성: 로컬에선 해석이 안 되는 내부 도메인(사내망 호스트)을 프록시가 있는 쪽 네트워크에서 해석해야 접속됩니다. 로컬 DNS로 하면 "그런 호스트 없음"이 뜨죠.

## ssh -D 1080 이 만드는 것

---

- `ssh -D 1080 user@host` 는 로컬 1080 포트에 SOCKS 프록시를 띄운다.
- 클라이언트가 localhost:1080 으로 SOCKS 요청을 보내면
    - SSH 가 암호화 터널로 서버에 전달하고
    - 서버의 sshd 가 목적지로 실제 연결을 대신 연다.

> 서버 쪽엔 sshd(SSH 데몬)만 있으면 됨 — 별도 프록시 SW 설치 불필요. "설치 없이 즉석 SOCKS"가 ssh -D의 핵심 매력.

## SOCKS 가 HTTP 프록시보다 범용인 이유

---

- `SOCKS` 는 내용을 안가리고 TCP 연결 자체를 대리하니까, HTTP 뿐 아니라 아무 프로토콜이나 태울 수 있다.
    - SSH/MySQL/Redis ...
- 반면 HTTP 프록시는 HTTP 를 이해해서 대리하는 구조라, HTTP 가 아닌 프로토콜은 CONNECT 터널로만 가능하고, 그마저 포트 제한에 걸린다.
