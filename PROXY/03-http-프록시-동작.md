# HTTP 프록시의 동작

---

## 평문 HTTP GET 을 프록시로 보낼 때,, 요청 라인이 어떻게 다른가?

---

### 직접 연결일 떄는 경로만 쓴다. (`origin-form`)

---

```text
GET /page.html HTTP/1.1
HOST: example.com
```

### 프록시 경유일 때는 전체 URL 을 쓴다. (`absolute-form`)

---

```text
GET http://example.com/page.html HTTP/1.1
HOST: example.com
```

### 왜?

---

- 직접 연결이면 이미 example.com 에 TCP 로 붙어있으니, "어느 서버" 인지 말할 필요가 없다.
- 하지만 프록시는 여러 목적지 중 어디로 보낼지 몰라서, 클라이언트가 요청 라인에 "이거 example.com 으로 보내줘" 라고 전체 주소를 명시한다.

## HTTPS(TLS) 는 프록시가 못 읽는데 어떻게 중개하나?

---

- HTTPS 는 클라이언트/서버 간 종단 암호화라, 중간의 프록시가 내용을 복호화 할 수 없다.
- 평문처럼 "GET http://..." 로 대리 요청을 할 수 없다. (암호화 되어있으므로 URL 도 못본다.)

그래서 `CONNECT` 메서드를 쓴다.

```text
CONNECT example.com:443 HTTP/1.1
Host: example.com
```

- 클라이언트가 프록시에게 "example.com:443" 까지 TCP 통로만 뚫어달라고 한다.
- "내용은 볼 것 없고, 그냥 양쪽으로 바이트만 전달해" 라고 요청한다.

**TLS 연결**

1. 프록시가 목적지와 TCP 연결을 하고, 성공한다.
2. 클라이언트는 해당 통로를 통해 프록시를 거쳐 서버와 직접 TLS 핸드셰이크를 한다.
3. 프록시는 암호문을 그대로 양방향 전달만한다.

```text
TCP:  클라 ═══════ 프록시 ═══════ 서버     ← TCP 2개 (클라-프록시, 프록시-서버)
TLS:  클라 ───────────────────── 서버     ← TLS 1개 (양 끝이 클라·서버)
```

## CONNECT 터널이 열린 뒤 프록시가 하는 일 (바이트 릴레이)

---
아무 해석도 안 하고, 받은 바이트를 반대쪽으로 그대로 복사만 한다.

- 클라이언트 ──(암호문)──▶ 프록시 ──(그대로)──▶ 서버
- 클라이언트 ◀──(그대로)── 프록시 ◀──(암호문)── 서버

프록시는 이 시점에 그냥 양방향 파이프이다. 내용이 HTTP 인지 뭔지도 모르고, TLS 암호문이라 알 수도 없다.

- 그냥 bytes in -> bytes out 만 한다.
- 이걸 `터널링(tunneling)` 이라고 한다.

## 프록시가 HTTPS 내용을 들여다보려면? (Man In The Middle)

---

정상적으로는 못본다. 보려면 `MITM` 방식을 써야한다.

1. 프록시가 클라이언트에게는 가짜 인증서(프록시가 자체 CA 로 즉석 발급) 를 내밀어 클라이언트와 TLS 를 맺는다.
2. 프록시가 진짜 서버와는 별도로 TLS 를 맺는다.
3. 프록시가 중간에서 복호화 -> 내용 확인 -> 재암호화

이게 성립하려면 **클라이언트가 프록시의 CA 인증서를 신뢰하도록 미리 설치해야한다.**

```text
TCP:  클라 ═══════ 프록시 ═══════ 서버     ← TCP 여전히 2개 (변화 없음!)
TLS:  클라 ─────── 프록시 ─────── 서버     ← TLS 2개 (클라-프록시, 프록시-서버)
```

## Via, X-Forwarded-For, Proxy-Connection 헤더

---

- `X-Forwarded-For (XFF)`
    - 프록시가 "원래 클라이언트 IP 는 이거였어" 라고 백엔드에 알려주는 헤더.
    - 프록시를 거치면 서버가 보는 출발지는 프록시 IP 가 되므로, 원 IP 를 잃지 않으려고 헤더에 실어 전달한다.
    - 여러 프록시를 거치면 콤마로 누적된다.
- `Via`
    - 요청/응답이 어떤 프록시들을 거쳤는지 경로를 기록하는 표준 헤더 (프록시 식별/루프 감지용)
- `Proxy-Connection`
    - 클라이언트 <-> 프록시 구간의 연결 유지(keep-alive) 제어용으로 쓰이던(비표준) 헤더
    - Connection 의 프록시 버전 격

## HTTP 프록시로는 왜 임의의 TCP 프로토콜을 그냥 못태우나?

---

- HTTP 프록시는 기본적으로 HTTP 요청을 이해해서 대리한다.
- DB 연결(예: MySQL 프로토콜) 은 HTTP 가 아니라서 "GET http://..." 식으로 대리 요청을 할 수 없다.
- 유일한 우회가 `CONNECT` 터널인데, 프록시는 보통 `CONNECT` 허용 포트를 443 정도로 제한한다.
- 그래서 3306(MySQL) 같은 포트로는 `CONNECT` 가 거부되는게 일반적이다.

## 캐싱 프록시가 "캐시해도 되는지" 를 판단할 때 보는 헤더

---

- `Cache-Control` (예: no-store, no-cache, max-age=3600, private/public)
- `Expires`
- `ETag`
- `Last-Modified`
- `Vary`

## 요약

---

| 상황               | 프록시가 하는 일                                |
|------------------|------------------------------------------|
| HTTP(평문)         | 요청 라인에 전체 URL받아 대리 요청. 내용 봄 -> 캐싱/필터링 가능 |
| HTTP(TLS)        | CONNECT 로 터널만 뚫고 바이트 릴레이. 내용 못봄          |
| HTTP(TLS) - 내용보기 | MITM(가짜 인증서 + 클라이언트 CA 신뢰 필요)            |
| 임의 TCP           | HTTP 프록시로는 어려움 -> SOCKS 필요               |
