# 프록시란 무엇인가

---

## 프록시를 한 문장으로 정의하면?

---

- 프록시란, 클라이언트와 목적지 서버 사이에서 요청을 대신 전달해주는 중개 서버(대리인) 이다.
- 클라이언트가 서버에 직접 붙지않고, "이 요청을 대신 처리해줘" 라고 프록시에 맡기면 프록시가 서버에 다녀와 응답을 돌려준다.

## 직접 연결 vs 프록시 경유 데이터 흐름

---

```text
[직접 연결]
클라이언트 ───────────────────▶ 서버
        ◀───────────────────
```

- 클라이언트가 서버와 직접 TCP 연결한다.

```text
[프록시 경유]
클라이언트 ──▶ 프록시 ──▶ 서버
        ◀──      ◀──
```

- 연결이 2개이다.
    - 클라이언트 <-> 프록시, 
    - 프록시 <-> 서버

## 프록시를 쓰는 대표 목적

---

1. **출발지 IP 변경/은닉** - 서버가 보는 클라이언트 IP 가 프록시 IP 가 된다. (익명화, IP 허용목록 통과)
2. **접근 제어/필터링/감사** - 특정 사이트 차단, 모든 요청 로깅 등. (회사 아웃바운드 프록시)
3. **캐싱** - 자주 요청되는 응답을 프록시가 저장해서 사용한다. 속도↑, 서버 부하↓.
4. **부하 분산, 서버 보호** - (`reverse`) 여러 백엔드 앞에서 트래픽 분배, TLS 종료, DDos 완충
5. **프로토콜 변환** - HTTPS 프록시가 받아 백엔드에는 HTTP 로 전달하는 등 기능이 가능
6. **관측/모니터링** - 트래픽을 한 지점에서 계측

## 출발지 IP 는 헤더를 고쳐서 못 바꾼다.

---

- IP 통신은 양방향이다.
- 내가 서버에 요청을 보내면, 서버는 그 패킷의 출발지(source IP) 로 응답을 보낸다.
- 그래서 출발지를 가짜로 적으면
    - 서버의 응답이 가짜 IP 로 가버린다.
    - TCP 는 3-way-handshake 로 시작하는데, SYN-ACK 를 받지못해 연결 자체가 안맺어진다.

## 그럼 프록시는 어떻게 IP 를 바꾸나?

---

**연결이 2개로 쪼개진다.**

- 클라이언트 <-> 프록시 : 클라이언트가 프록시에 연결한다.
- 프록시 <-> 서버 : 프록시가 자신의 IP 로 새 TCP 연결을 연다.

## 프록시 도입의 단점/비용

- **지연(latency) 증가** - 홉이 하나 늘어난다.
- **단일 장애점(SPoF)** : 프록시가 죽으면 그 경로의 모든 트래픽이 끊기게 된다.
- **병목/확장비용** : 모든 트래픽이 한 지점을 통과하게된다. -> 대역폭/CPU 한계, 스케일링이 필요하다.
- **신뢰 경계 문제** : 프록시는 트래픽을 볼 수 있다. 프록시가 탈취되면 위험하다.
- **운영 복잡도** : 설정, 인증서, 접근통제, 모니터링 대상이 하나 더 늘어난다.
- **디버깅 난이도** : 문제가 클라/프록시/서버 중 어디서 났는지 층이 늘어 원인 추적이 복잡하다.
