# 초기적재와 스냅샷

---

- binlog 는 처음부터 전체 로그를 갖고있지 않다. 주기적으로 purge 되기 때문이다.
- 이미 데이터가 들어있는 DB 에 CDC 를 붙일 때, binlog 를 처음부터 재생할 수는 없다.
- 이를 `스냅샷` 방식으로 해결한다.

## initial snapshot

---

**방식**

1. **모든 테이블**에 대해서 SELECT 를 사용해 통째로 읽는다.
    - `op=r` 이벤트로 발행한다.
2. 이 시점의 binlog 위치를 기록한다. (`baseline`)
3. `baseline` 부터 스트리밍을 시작한다.

**baseline 을 잡는법**

1. 글로벌 읽기 락을 건다. (FLUSH TABLES WITH READ LOCK)
2. 현재 binlog 위치를 기록한다.
3. REPEATABLE READ 트랜잭션을 시작하여 데이터를 읽는다. (일관된 데이터를 읽을 수 있다.)
4. 글로벌 읽기 락을 해제한다.
5. 락 없이 MVCC 로 모든 테이블의 데이터를 읽는다.
6. 이후 읽기가 끝나면 `baseline` 부터 스트리밍을 재개한다.

## initial snapshot 의 단점

---

1. 락 문제
    - baseline 을 잡기위해 잠시동안 글로벌 읽기 락(FLUSH TABLES WITH READ LOCK) 을 사용한다.
    - 운영 DB 에는 치명적이다.
2. 긴 지연
    - 스냅샷에 대해 전부 처리해야 스트리밍이 시작 가능했다.
    - 큰 테이블이면 몇시간 이상 지연될 수 있다.
3. 중단 취약
    - 스냅샷 처리 중 실패하면 처음부터 다시 시작해야한다.
4. 테이블 추가의 어려움
    - initial snapshot 방식은 특정 테이블만 스냅샷을 뜰 방법이 없었다.
    - 이로인해 테이블을 추가하기위해서는 전체 스냅샷을 다시 돌려야했다.

## incremental snapshot

---

- 테이블을 PK 순서로 chunk(기본 1024행) 씩 끊어 읽는다.
    - SELECT ... WHERE pk > :마지막읽은pk ORDER BY pk LIMIT 1024
- 이걸 반복하며, 그 사이 binlog 스트리밍도 계속 돈다.

| initial 의 문제 | incremental 이 푸는 방식            |
|--------------|--------------------------------|
| 락 (FTWRL)    | chunk 마다 짧은 SELECT 뿐, 글로벌 락 없음 |
| 긴 지연         | 스트리밍이 안 멈춤 (동시 진행)             |
| 중단 취약        | 죽으면 마지막 pk 부터 재개 (처음부터 X)      |
| 테이블 추가       | signal 로 그 테이블만 청크 스냅샷         |

## incremental snapshot 핵심 난제 - watermark 알고리즘

---

상황

```text
users 테이블을 chunk 로 스냅샷 중. 이번 chunk 가 읽을 대상 : pk = 1, 2, 3.
동시에 다른 사용자가 앱에서 pk=2 를 수정하고 있다.
```

`binlog` 는 아래의 시간순으로 흐른다.

```text
...(이전 이벤트들)
[low 말뚝 박음]           ← ① 내가 "지금부터 chunk 읽기 시작" 표시
UPDATE pk=2 -> "김철수"   ← 앱에서 들어온 실제 변경 (내 SELECT와 겹침!)
[high 말뚝 박음]          ← ③ 내가 "chunk 읽기 끝" 표시
```

그리고 ② 내 SELECT 결과는 이랬다. (low ~ high 말뚝 사이 어딘가에서 읽음)

```text
pk=1 → "이영희"
pk=2 → "김OO"     ← 이 값이 수정 전인지 후인지 애매함! (동시에 벌어짐)
pk=3 → "박민수"
```

**최종 발행**

```text
op=r  pk=1  → "이영희"    ✅ (아무도 안 건드림 → 스냅샷 값 신뢰)
op=r  pk=3  → "박민수"    ✅ (아무도 안 건드림 → 스냅샷 값 신뢰)
op=u  pk=2  → "김철수"    ← 스트리밍이 이미/따로 보내줌 (최신값!)
```

`watermark` 는

- "내가 chunk 를 읽는 동안 바뀐 행" 을 골라내는 말뚝이다.
- 걸린 행은 스냅샷에서 빼고 스트리밍에 맡긴다.

## 번외 - signal

---

**개념**

- Debezium 한테 "이 작업 좀 해줘" 라고 지시를 넣는 통로이다.
- 커넥터를 재시작하거나 설정을 바꾸지 않고, 실행 중에 명령을 꽂아넣는 방법이다.

`signaling table` 방식

- 가장 흔한 방식이며, DB 에 전용 테이블을 하나 두는 것이다.
- 여기에 INSERT 를 하면 그게 명령이 된다.
    - 이 INSERT 도 debezium 이 binlog 로 읽게되어 처리된다. - 별도 API/포트가 필요하지않다.

```sql
CREATE TABLE debezium_signal
(
    id   VARCHAR(64) PRIMARY KEY,
    type VARCHAR(32),
    data VARCHAR(2048)
);

-- shop.users 테이블 incremental snapshot 을 떠라 는 명령 --
INSERT INTO debezium_signal (id, type, data)
VALUES ('ad-hoc-1', 'execute-snapshot', '{"data-collections": ["shop.users"]}')
```

대표적인 signal 명령들

- execute-snapshot : ad-hoc/incremental 스냅샷 시작
- stop-snapshot : 진행 중 스냅샷 중단
- pause-snapshot / resume-snapshot : 일시정지/재개
- log : 로그 찍기 (디버깅용)
