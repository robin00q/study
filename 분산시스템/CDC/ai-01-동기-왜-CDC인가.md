# 전통적인 방식 (ETL)

---

**개념**

- Extract(원본에서 뽑고) -> Transform(가공하고) -> Load(목적지에 적재) 한다.
- 전통적인 배치로, 보통 하루 한 번 같이 대량으로 수행한다.
- **"updated_at 기준으로 N주기 마다 폴링"** 하는 방식도 사용할 수 있다.

**query-based 의 한계**

1. 삭제(DELETE) 를 잡을 수 없다. - 행이 사라지면 폴링으로는 알 수 없다.
2. 중간 변경이 누락된다. - 1시간 안에 A->B->C 로 3번 바뀌면 폴링은 C 밖에 알 수 없다.
3. 실시간이 아니다. - 최대 1시간 지연된다. 검색/캐시에는 너무 느리다.
4. updated_at 을 신뢰할 수 있어야한다. - 개발과정에서 updated_at 을 누락한다면 문제가 생길 수 있다.

특히 **1번(삭제 누락)** 이 결정타이다. log-based CDC 는 binlog 에 DELETE 이벤트로 이를 처리 할 수 있다.

## 핵심 개념 (내 말로)

- 이기종 간의 동기화를 하기 위해서는 보통 비동기방식으로 이를 동기화한다.
- 주로 MySQL 같은 관계형 DB 가 source of truth 로 관리되며, 여기서 변경된 데이터를 다른 시스템에 동기화하기위해 사용한다.
- Debezium 같은 툴을 사용하여 mysql binlog 는 변경된 이력이 모두 관리되니, 해당 로그를 사용해서 이벤트를 kafka 에 발행하고 eventually consistent 하게 처리하게 한다.
