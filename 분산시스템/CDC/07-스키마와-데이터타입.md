# 스키마와 데이터 타입

---

## 원본 -> Debezium : DDL 을 어떻게 감지하나

---

- MySQL binlog 에는 row 변경뿐 아니라, `DDL` 도 기록된다.
- Debezium 은 이 DDL 을 파싱해서 내부 스키마 모델을 갱신하고
    1. **debezium 이 사용하려고** - 메모리에 기록
    2. **재시작 복구용** - schema.history.internal 토픽에 append 한다.
    3. **스키마 변경 토픽(<topic.prefix>)** 에 발행 - 다운스트림(데이터 웨어하우스, 앱... 등) 공지용

```json
{                                                                                                                                                                                                        
  "databaseName": "inventory",                                                                                                                                                                           
  "ddl": "ALTER TABLE customers ADD COLUMN age INT",                                                                                                                          
  "tableChanges": [                                                                                                                                                           
    {                                                                                                                                                                                                    
      "type": "ALTER",                                                                                                                                                                                   
      "id": "inventory.customers",                                                                                                                                                                       
      "table": {                                                                                                                                                                                         
        "primaryKeyColumnNames": ["id"],                                                                                                                                                                 
        "columns": [ {"name":"id",...}, {"name":"age",...} ]                                                                                                                                             
      }                                                                                                                                                                                                  
    }                                                                                                                                                                                                    
  ],                                                                                                                                                                                                     
  "source": { ... },                                                                                                                                                                                     
  "ts_ms": 123456789                                                                                                                                                                                     
}
```

## Debezium -> Kafka : 이벤트 스키마가 어떻게 바뀌나

---

- Debezium 이벤트에는 payload(데이터 - 값) 과 그 구조(schema) 가 같이 발행된다.
- `ALTER TABLE customers ADD COLUMN age INT` 를 하면
    - ALTER 이전 이벤트 : 옛 스키마 그대로
    - ALTER 이후 이벤트 : payload, schema 에 age 필드가 포함된다.
- 그래서 한 토픽에 **서로 다른 버전의 스키마를 가진 메시지가 섞여** 있게 된다.
- Debezium 은 각 이벤트를 **"그 이벤트가 발생한 시점의 스키마"** 로 직렬화하여 메시지를 producing 한다.

## Kafka -> 컨슈머 : 안깨지려면 (호환성 & Schema Registry)

---

- 컨슈머가 예전/새로운 스키마 섞인 데이터를 안전하게 읽으려면 스키마 호환성 규칙이 필요하다.
- **Schema Registry** - 스키마를 중앙에 저장 및 버전을 관리하는 서비스이다. 메시지에는 전체 스키마 대신 **스키마 ID(정수)** 만 실어 경량화한다.
