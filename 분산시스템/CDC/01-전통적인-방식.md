# 전통적인 CDC 방식

---

## query-based (ETL)

---

개념

- Extract(원본에서 뽑고) -> Transform(가공하고) -> Load(목적지에 적재) 한다.
- 전통적인 배치로, 보통 하루 한 번 같이 대량으로 수행한다.
- **"updated_at 기준으로 N주기 마다 폴링"** 하는 방식도 사용할 수 있다.

**query-based 의 한계**

1. 삭제(DELETE) 를 잡을 수 없다. - 행이 사라지면 폴링으로는 알 수 없다.
2. 중간 변경이 누락된다. - 1시간 안에 A->B->C 로 3번 바뀌면 폴링은 C 밖에 알 수 없다.
3. 실시간이 아니다. - 최대 1시간 지연된다. 검색/캐시에는 너무 느리다.
4. updated_at 을 신뢰할 수 있어야한다. - 개발과정에서 updated_at 을 누락한다면 문제가 생길 수 있다.

특히 **1번(삭제 누락)** 이 결정타이다. log-based CDC 는 binlog 에 DELETE 이벤트로 이를 처리 할 수 있다.

## trigger-based

---

트리거 : INSERT/UPDATE/DELETE 가 일어날 때, DB 가 자동 실행하는 내장 함수이다.

개념

- 테이블마다 변경이 일어나면 DB 의 트리거를 걸어놓는다.
- 행이 바뀔때마다 트리거가 별도의 `변경 이력 테이블(shadow/audit table)` 에 그 변경을 기록한다.
- CDC 는 그 이력 테이블을 읽어서 전파한다.

처리방식

1. 플래그 방식 - audit 테이블에 `processed=false` 하고, 주기적으로 어플리케이션이 처리한 뒤 `processed=true` 로 업데이트한다.
2. 워터마크 방식
    - 어플리케이션이 마지막으로 읽은 id 를 기억하고, WHERE id > 마지막_id 만 읽는다.
3. 처리한 행은 삭제하는 방식 - 재처리가 불가능하다.

**trigger-based 의 한계**

1. intrusive - 트리거가 원본 트랜잭션 안에서 실행된다. 
    - 매 쓰기마다 추가 INSERT 가 붙는다. 
    - 트리거 오류가 본 트랜잭션에 영향을 준다. 
2. 운영부담 - 테이블마다 트리거를 만들어 관리해야한다.
