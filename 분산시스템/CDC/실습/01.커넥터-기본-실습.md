# 커넥터 기본 실습

---

## 목표

---

- 등록된 커넥터가 실제로 INSERT/UPDATE/DELETE 를 어떻게 이벤트로 흘리는지 확인한다.

## 인스턴스 들여다보기

---

### GET /connectors

---

```shell
> curl localhost:8083/connectors
["inventory-connector"]
```

- 이 Connect 에 접수된 커넥터는 단 하나. inventory-connector 이다.

### GET /connectors/{connector-name}/status

---

```shell
> curl -s localhost:8083/connectors/inventory-connector/status | jq
{
  "name": "inventory-connector",
  "connector": {
    "state": "RUNNING",
    "worker_id": "172.24.0.5:8083"
  },
  "tasks": [
    {
      "id": 0,
      "state": "RUNNING",
      "worker_id": "172.24.0.5:8083"
    }
  ],
  "type": "source"
}
```

- connectors.state : 감독이 살아있나 (설정 읽고 task 배치하는 관리 역할)
- tasks[].state : 실제로 MySQL binlog 를 빨아들이는 일꾼이 살아있나
- type : 커넥터의 종류. source/sink

실무에서는
- connector=RUNNING 인데, task=FAILED 인 경우가 흔하다.
- 그러면 이벤트가 흐르지 않는다. 따라서 task 상태까지 잘 봐야한다.

### GET /connectors/{connector-name}/config

---

```shell
> curl -s localhost:8083/connectors/inventory-connector/config | jq
{
  "connector.class": "io.debezium.connector.mysql.MySqlConnector",
  "database.user": "debezium",
  "database.server.id": "184054",
  "incremental.snapshot.chunk.size": "1",
  "tasks.max": "1",
  "schema.history.internal.kafka.bootstrap.servers": "kafka:9092",
  "database.port": "3306",
  "topic.prefix": "dbserver1",
  "schema.history.internal.kafka.topic": "schemahistory.inventory",
  "database.hostname": "mysql",
  "database.password": "dbz",
  "signal.data.collection": "inventory.debezium_signal",
  "name": "inventory-connector",
  "database.include.list": "inventory"
}
```

**어떤 커넥터인가?**

- name: 이 인스턴스의 이름
- connector.class : `MySQL` 용 Debezium 플러그인으로 떠라 (Mongo 면 Mongo 로, Postgres 면 Postgres 로)

**어디에 어떤 계정으로 붙나?**

- database.hostname, database.port, database.user, database.password : 붙을 DB 관련 정보. **binlog 를 읽을 복제권한**이 있어야한다.

**가짜 replica 훙내내기**

- database.server.id : Debezium 이 가짜 replica 로 등록될 때 쓰는 서버 ID. MySQL 복제에서는 모든 replica 가 고유한 server_id 를 가져야한다.

**무엇을 캡처하나 (범위)**

- database.include.list : 여기서는 `inventory` DB 만 캡쳐한다. 더 줄이면 `table.include.list` 로 선언할 수 있다.

**스키마 변경 이력 저장소 (MySQL 특유)**

- schema.history.internal.kafka.bootstrap.servers :  
- schema.history.internal.kafka.topic :

**incremental snapshot**

- signal.data.collection : signal 대상 테이블
- incremental.snapshot.chunk.size : incremental-snapshot 시 읽을 데이터의 chunk size (기본은 1024 일 것이다.)

## 커넥터 생성/삭제 (생명주기)

---

### POST /connectors

---

```shell
> curl -s -X POST http://localhost:8083/connectors \
    -H "Content-Type: application/json" \
    -d '{
      "name": "inventory-connector",
      "config": {
        "connector.class": "io.debezium.connector.mysql.MySqlConnector",
        "tasks.max": "1",
        "database.hostname": "mysql",
        "database.port": "3306",
        "database.user": "debezium",
        "database.password": "dbz",
        "database.server.id": "184054",
        "topic.prefix": "dbserver1",
        "database.include.list": "inventory",
        "schema.history.internal.kafka.bootstrap.servers": "kafka:9092",
        "schema.history.internal.kafka.topic": "schemahistory.inventory",
        "signal.data.collection": "inventory.debezium_signal",
        "incremental.snapshot.chunk.size": "1"
      }
    }'
```

### DELETE /connectors/{connector-name}

---

```shell
> curl -s -X DELETE http://localhost:8083/connectors/inventory-connector
```

- 등록된 커넥터를 삭제한다.
