# incremental snapshot 실습 - signal & watermark 관찰

---

## 트리거 - signal INSERT

---

- `debezium_signal` 테이블에 INSERT 하면 스냅샷을 실행하라는 명령을 할 수 있다.
    - DB 에 INSERT 되면 binlog 를 타고 debezium 에 전달된다.

```mysql
INSERT INTO debezium_signal (id, type, data)
VALUES ('r-1', 'execute-snapshot', '{"data-collections": ["inventory.customers"], "type": "incremental"}');
```

## 겪은 문제 - watermark 쓰기 권한

---

- signal 신호는 전달됐으나, debezium 이 `debezium_signal` 테이블에 기록하는데 실패한 이슈이다.
    - signal 신호를 접수하면 debezium 은 watermark 를 `debezium_signal` 테이블에 기록하고 사용하기 때문
        - watermark 를 찍는다는 것은 **signal 테이블에 snapshot-window-open/-close 행을 INSERT 하는 것**

```mysql
GRANT INSERT, UPDATE, DELETE ON inventory.debezium_signal TO 'debezium'@'%';
```

## 결과 관찰

---

- customers 토픽에 op=r, source.snapshot=incremental
- chunk.size=1 -> 1001~1005 를 한 건 씩
- 같은 행이 여러번 재발행되면 -> key = PK 라 upsert 로 멱등적이다.

**low/high watermark**

- signal 테이블을 조회시 <uuid>-open / <uuid>-close (type = snapshot-window-open/close) 행이 청크마다 생긴다.
- watermark 행에는 **timestamp** 만 있고, "어떤 행인지" 는 없다.
    - 행 식별은 connect_offsets 의 (incremental_snapshot_primary_key/maximum_key) + 커넥터 메모리에서 관리된다.
