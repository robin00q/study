# 트랜잭션 로그 (DB 기본기)

---

## DB 는 왜 트랜잭션 로그를 사용할까? - WAL

---

**WAL(Write-Ahead Logging) 원칙**

- 데이터가 파일을 직접 고치기 전에, 변경을 로그에 먼저 적는다.
- 데이터가 파일 **1. 여기저기를 랜덤하게 쓰는것은 느리고**, **2. 쓰다가 종료되면 데이터가 깨진다.**
- 반면 로그는 끝에 **이어붙이기(append) 만 하므로 빠르다.**
- `벌크반영` - 데이터 파일은 나중에 모아서 반영한다.
- `장애복구` - 종료 후 살아나면 로그를 재생(replay)해서 복구한다.

`WAL` 에서 커밋이 됐다는 것은 아래를 의미한다.

- ✅로그 레코드가 디스크에 안전하게 쓰인(`fsync`) 순간이다. - 나중에 체크포인트에 DB 에 반영된다.
- ❌DB 에 row 로 기록됐다.

이걸 구현한것이 `redo log` 이다.


## redo log - 나(DB) 를 위한 복구용 로그

---

데이터 파일과 메모리의 "최근 차이" 를 메꾸기위한 로그이다.

- 목적 : 크래시 복구 및 벌크 write
- 커밋 : redo 가 fsync 된 순간

DB 는 디스크에 있는 데이터와 로그를 적용한 결과물이다.

## binlog - replica, CDC 를 위한 전달용 로그

---

데이터의 변경분을 전달하기 위한 로그이다.

- 목적 : 복제, CDC
- 포맷 : statement / row / mixed
- 일정 보존기간 이후 정리한다.

**MySQL binlog 포맷 (statement / row / mixed)**

- `statement` : 실행된 SQL 문 자체를 기록한다. (UPDATE t set x=1 WHERE ...)
- `row` : 실제 바뀐 행의 before/after 이미지를 기록한다.
- `mixed` : 상황 따라 둘을 자동으로 선택한다.

**CDC 에는 `row` 를 사용한다.**

- `statement` 기반은 "재현하면 같은 결과" 라는 보장이 약하다.
    - NOW(), RAND(), AUTO_INCREMENT 같은 비결정적 요소가 있으면 SQL 로 무슨 값이 들어갔는지 알 수 없다.
- `row` 기반은 행이 실제로 이렇게-> 저렇게 바뀌었다는 확정된 사실을 담는다.

**순서 보장**

- 로그는 커밋된 순서대로 끝에 `append-only` 만 한다. 무엇이 먼지 일어났나에 대한 모호함이 없다.
- auto-increment id 는 `발급 순서` 라 `커밋 순서`와 어긋나지만 로그는 `커밋 순서` 로 기록된다.
