**서로 다른 DB, 서로 다른 데이터센터, 서로 다른 클라우드에 흩어진 데이터를 일관되게 유지하는 문제다.**

### 동기화 방식

---

#### (A) 타이밍

|  | 동기 | 비동기 |
| --- | --- | --- |
| 일관성 | 강함(strong) | 최종적(eventual) |
| 지연 | 느림(쓰기가 모든 곳에 반영될 때까지 대기) | 빠름 |
| 가용성 | 낮음(한 곳 죽으면 전체 막힘) | 높음 |
| 현실 사용 | 금융 원장 등 일부 | 이기종 싱크의 대부분 |
 
- 이기종 환경에서는 시스템마다 특성이 너무 달라서 동기 방식은 거의 불가능하다.
- **비동기 + 최종 일관성(eventual consistency)** 가 사실상 표준이다.

#### (B) 동기화 단위

- 전체 스냅샷 : 주기적으로 통째로 복사하여 동기화한다. 단순하지만 무겁고 느리다.
- 증분/변경분(CDC) : 바뀐 것만 전파한다. 효율적이지만 구현이 복잡하다.

### 핵심기법 : CDC (Change Data Capture)

---

이기종 동기화의 주류 패턴이다. 깊게 공부할 필요가 있다.

대표 도구

- `Debezium` (Kafka 기반 CDC 의 사실상 표준이다.)
- [MySQL binlog] → [Debezium] → [Kafka] → [Sink Connectors] → [ES / Redis …]

### 신뢰성을 위한 필수 패턴들

---

이기종 싱크에서 데이터가 “유실” 되거나 “어긋나는” 걸 막는 패턴들이다.

`Outbox 패턴`

- 문제 : DB 에 저장 + 메시지를 큐에 발행을 둘 다 해야하는데, **하나만 성공하면 데이터가 어긋**난다. (`dual-write problem`)
- 해결 : 비즈니스 데이터와 “보낼 이벤트” 를 같은 트랜잭션으로 outbox 테이블에 저장한다. → CDC 가 outbox 테이블을 읽어 발행한다. (원자성 보장)

`멱등성 (Idempotency)`

- 비동기 시스템은 “최소 한 번 (at-least-once)” 전달이 기본이라 같은 메시지가 두 번 올 수 있다.
- 따라서 소비자는 같은 변경을 두 번 받아도 결과가 같아야한다.

`순서 보장(Ordering)`

- 같은 키(예: user_id) 의 변경은 순서가 뒤바뀌면 안된다.
- Kafka 는 파티션 단위로 순서를 보장하므로 보통 키로 파티셔닝한다.

### 검증

---

정말 싱크가 맞았는지 어떻게 아는가?

싱크를 맞추는 것만큼 싱크가 맞았는지 확인하는 것도 중요하다.

- `Reconciliation` : 양쪽 시스템의 row count, 체크섬, 샘플 비교를 주기적으로 수행한다.
- `Merkle Tree` : 대량 데이터를 효율적으로 비교한다. 다른 부분만 빠르게 찾아낸다.
